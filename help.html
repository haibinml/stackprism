StackPrism 使用说明

这页专门讲怎么添加自己的识别规则。你可以把规则理解成一句话：看到什么字，就认为网页用了什么东西。

先知道它在做什么

插件会看网页里出现的线索。比如网页代码里有 wp-content ，大概率就是 WordPress；网页加载了 hm.baidu.com ，大概率用了百度统计。

你添加规则，就是告诉插件：“以后看到这个字、这个网址、这个标签，就显示我指定的技术名称”。

最快上手：五步添加一条规则

  1. 打开设置页，找到“自定义规则”。
  2. 在“技术名称”里写你想显示的名字，比如 MyCMS 。
  3. 在“分类”里选一个大类，比如 网站程序 。
  4. 在“匹配规则”里写网页里会出现的字，一行写一个，比如 mycms 。
  5. 点“添加规则”，再点页面右上角“保存设置”。回到目标网页刷新一下，就能测试。

每个输入框怎么填

技术名称

最终显示在插件里的名字。比如 ExampleCMS 、 某某统计 、 某某支付 。

分类

放到哪个分组里。不会影响匹配，只影响结果显示在哪个 Tab 下。

类型说明

给自己看的补充说明。比如 CMS 、 第三方支付 、 开源统计 。

置信度

你有多确定。只要看到就能确定，选“高”；只是猜测，选“低”；拿不准就选“中”。

匹配方式

新手建议先选“关键词”。如果你会写更灵活的规则，再选“正则表达式”。

官网 / 仓库 URL

填了以后，检测结果里的技术名称可以点开官网或项目地址。没有就留空。

匹配范围怎么选

不用纠结，刚开始可以全选。等规则误报太多，再回来减少范围。

页面 URL

看当前网页地址。比如地址里出现 /shop/ 。

资源 URL

看网页加载的脚本、图片、样式地址。比如 cdn.example.com/sdk.js 。

DOM / 源码

看网页最终显示出来的代码。适合找 data-xxx 、隐藏标签、页面里的特殊字。

响应头

看服务器返回的信息。比如 X-Generator: WordPress 。

动态资源

看你操作网页后新加载出来的东西。适合弹窗、登录、支付、聊天组件。

示例 1：识别一个网站程序

假设你发现某个网站程序叫 MyCMS，它的网页里经常出现 mycms 或 X-Generator: MyCMS 。

技术名称
MyCMS
分类
网站程序
类型说明
CMS
匹配方式
关键词
匹配规则
mycms
X-Generator: MyCMS

解释：以后网页里只要出现 mycms 这个字，插件就会显示 MyCMS。

示例 2：识别 WordPress 主题

很多 WordPress 主题的地址长这样： /wp-content/themes/主题名/ 。如果你只想识别某个主题，可以这样写。

技术名称
WordPress 主题: oceanwp
分类
主题 / 模板
匹配方式
关键词
匹配规则
/wp-content/themes/oceanwp/

解释：看到这个主题目录，就显示 WordPress 主题: oceanwp 。

示例 3：识别统计脚本

如果某个统计服务会加载 stats.example.com/tracker.js ，可以用加载地址来识别。

技术名称
Example Stats
分类
统计 / 分析
类型说明
统计平台
匹配方式
关键词
匹配规则
stats.example.com
tracker.js

提醒： tracker.js 这个名字比较常见，可能误报。更稳的写法是只写 stats.example.com 。

示例 4：识别第三方登录或支付

登录和支付通常是点按钮后才加载，所以记得勾选“动态资源”。

技术名称
ExamplePay
分类
支付系统
类型说明
第三方支付
匹配方式
关键词
匹配规则
pay.example.com
examplepay-sdk

测试方法：保存规则后，刷新网页，再点一次网页里的支付按钮，然后重新打开插件看结果。

示例 5：用正则写更灵活的规则

正则可以理解成“带通配的关键词”。比如有些文件名会带版本号： examplecms-1.2.3.js 、 examplecms-2.0.0.js 。

匹配方式
正则表达式
匹配规则
examplecms-[0-9.]+\.js

大白话解释： [0-9.] 表示数字和点， + 表示前面的东西可以出现很多次。

怎么减少误报

  • 不要只写太普通的词，比如 app 、 main 、 login ，这些很多网站都有。
  • 优先写完整域名，比如 pay.example.com 比 pay 稳很多。
  • 优先写特殊目录，比如 /wp-content/themes/oceanwp/ 比 oceanwp 稳。
  • 不确定就把置信度设为“低”，这样看结果时会知道它只是猜测。
  • 规则保存后一定要刷新目标网页，再打开插件看结果。

想把规则贡献给项目

如果你的规则比较通用，可以回到设置页点击“提交规则贡献”。它会打开 GitHub 表单，你把技术名称、分类、匹配规则和证据填进去就行。

如果你会改仓库文件，内置规则放在 rules/ 目录。页面里能看到的字、脚本地址、主题目录这些，通常放到 rules/page/ ；响应头、Cookie、Server 这些，通常放到 rules/headers/ 。

证据可以是官网说明、测试网页、脚本地址、响应头截图或源码片段。敏感网址可以打码。
